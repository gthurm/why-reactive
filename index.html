Cargo cult programming

"Cargo cult programming is a style of computer programming characterized by the ritual inclusion of code or program structures that serve no real purpose. Cargo cult programming is typically symptomatic of a programmer not understanding either a bug they were attempting to solve or the apparent solution (compare shotgun debugging, deep magic)1".

1https://en.wikipedia.org/wiki/Cargo_cult_programming

Cargo cult programming existing members:

  • Agile methodologies - Scrum, Kanban
  • TDD - Test Driven Development
  • BDD - Cucumber, etc
  • OOP Design patterns
  • Functional Programming
  • Reactive programming

All can go pear shaped.

Cargo cult programming

  • Agile - cool but where is documentation and strategy to repay tech debt? Architecture?
  • TDD - unit tests will check your refactored code! Try to refactor bigger code base structurally with unit tests!
  • BDD - product owner will write cucumber tests - good one
  • OOP design patterns, what about ECS pattern, FP patterns?
  • RP - what about small little systems / apps like MobLab, what if there are high critical requirements on consistency? Life support systems?

What is this?

"In computing, reactive programming is a programming paradigm oriented around data flows and the propagation of change. This means that it should be possible to express static or dynamic data flows with ease in the programming languages used, and that the underlying execution model will automatically propagate changes through the data flow.1"

1https://en.wikipedia.org/wiki/Reactive_programming

What is this?

  • M$ Excel is an example of this paradigm in action. An update to one field can correspond to an update to one or more other fields.
  • Reactive programming grew way more out of that to much more than that - http://www.reactivemanifesto.org

Reactive Manifesto 2.0 - growth

  • Responsive - system responds in timely manner
  • Resilient - system stays responsive in case of failure
  • Elastic - system stays responsive under high load
  • Message Driven - reactive systems rely on async communication between system components

DevOps loves Reactive

  • In the past, they had to restart servers manually in case of network failures
  • Should all threads be blocked (out of memory under heavy load) - tanuki wrapper restarted the app
  • Reactive systems generally heal themselves unless there is a serious code bug

QAs hate Reactive

  • Applications need extra testing in case of failures
  • Manual or automated (Chaos Monkey)

What we already know.

  • CRUD - Create, Read, Update and Delete
  • ACID - ATOMICITY, Consistency, Isolation, Durability
  • BASE - Basically Available, Soft-state, Eventually Consistent

Reactive inspirations

  • Fallback for optional data - in case of system failure
  • Hot/cold endpoints
  • Circuit breakers (for hot endpoints only)
  • Bulkhead isolation pattern
  • Backpressure handling
  • Event Sourcing / CQRS

Global State

Set data-state="something" on a slide and "something" will be added as a class to the document element when the slide is open. This lets you apply broader style changes, like switching the page background.

State Events

Additionally custom events can be triggered on a per slide basis by binding to the data-state name. 


Reveal.addEventListener( 'customevent', function() {
	console.log( '"customevent" has fired' );
} );
